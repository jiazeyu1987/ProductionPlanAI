# LLM 重构执行手册（交接版）

更新时间：2026-03-28  
适用范围：新加入的 LLM 或新团队首次接手 `code/frontend` + `code/backend` 重构任务

## 1. 目标

本手册用于回答两个问题：

1. 新 LLM 如何在不破坏业务的前提下快速接手。
2. 新 LLM 如何按“每次重构都先测试通过”的规则持续推进。

## 2. 接手后必读输入

1. 流程基线：`01_当前业务流程梳理.md`、`02_当前数据流程梳理.md`
2. 璞慧生产独立基线：`03_璞慧生产_业务流程梳理.md`、`04_璞慧生产_数据流程梳理.md`
3. 并行重构主方案：`05_并行团队重构总方案.md`、`06_模块拆分与团队分工.md`
4. 质量闸门与集成：`07_分步重构与测试闸门.md`、`08_并行集成计划与风险清单.md`
5. 文件级任务映射：`10_模块文件映射与任务清单.md`
6. 契约治理：`11_契约冻结与变更流程.md`

## 3. 硬约束（不可破坏）

1. 重构阶段必须保持现有主流程可用：订单池 -> 排产 -> 发布/回滚 -> 报工 -> 告警/重排。
2. `/v1/*` 与 `/internal/v1/internal/*` 路由必须继续满足 Bearer 鉴权规则。
3. 所有写接口必须继续支持 `request_id` 幂等语义。
4. 每次只做一个小步改造（单模块、单责任、单风险）。
5. 每次小步改造完成后必须通过该步测试，才允许进入下一步。
6. 不将 `code/legacy-node` 纳入本轮重构目标。

## 4. 启动检查清单（首次接手）

1. 读取本目录文档并确认当前所在 Wave（见 `08`）。
2. 执行基线验证命令并记录结果：
   1. `cd code/backend && mvn test`
   2. `cd code/frontend && npm test`
   3. `cd code/frontend && npm run build`
   4. `cd code && powershell -ExecutionPolicy Bypass -File .\\scripts\\run_verify.ps1`
3. 在 `doc/refactor/test-records/` 新增一条“基线记录”。
4. 从 `10` 中选择一个模块的第一批任务，不跨模块混做。

## 5. 单步执行标准（每次都按此循环）

1. 选一个任务（例如 `BE-A-01`）。
2. 明确本步只允许修改的文件集合（避免漂移）。
3. 实施改造（结构拆分优先，不改变对外契约）。
4. 执行模块最小测试 + 跨模块最小回归。
5. 记录测试结果与回滚方案（`doc/refactor/test-records/`）。
6. 通过后再进入下一步；失败则先回滚或修复。

## 6. 输出物要求（每个 PR 必须具备）

1. 变更范围：改了哪些文件、属于哪个模块。
2. 契约影响：是否改动 API 入参/出参/错误码。
3. 测试证据：执行了哪些测试，结果如何。
4. 回滚方式：代码回滚、配置回滚、数据回滚说明。
5. 文档同步：至少更新模块任务状态与测试记录。

## 7. 接口变更特殊规则

1. 若改动仅限内部实现，不得修改既有接口行为。
2. 若必须改契约，先走 `11_契约冻结与变更流程.md` 的分级流程。
3. 未完成契约评审前，不得合并破坏性变更。

## 9. 判定“是否足够可重构”的标准

满足以下条件即判定“文档足够支撑新 LLM 重构”：

1. 有业务与数据基线（`01~04`）。
2. 有模块与团队边界（`05~06`）。
3. 有强制测试闸门（`07`）。
4. 有并行集成与风险控制（`08`）。
5. 有文件级任务映射与契约变更流程（`10~11`）。
