# 模块文件映射与任务清单（LLM 可执行）

更新时间：2026-03-28  
目的：把“模块拆分”落到文件级任务，便于新 LLM 直接执行。

## 1. 任务编号规则

1. 后端任务：`BE-团队字母-序号`，例如 `BE-A-01`
2. 前端任务：`FE-团队字母-序号`，例如 `FE-C-02`
3. 平台/治理公共任务：`PLAT-序号`

## 2. 阶段 0（并行前置，必须先完成）

当前后端核心逻辑高度集中在 `MvpStoreService.java`，直接并行改造会产生高冲突。  
因此先执行以下“并行前置”任务：

1. `PLAT-01`：建立目标目录骨架（见 `06`），不改业务逻辑。
2. `PLAT-02`：为各模块创建门面服务（Facade）与接口壳，Controller 先改为调用门面。
3. `PLAT-03`：将 `MvpStoreService` 变为“临时编排器”，逐步委托到模块服务。
4. `PLAT-04`：建立模块测试入口（每个模块最少 1 个 smoke）。

## 5. 冲突控制规则（必须执行）

1. `MvpStoreService.java` 在阶段 0 前由 Team-G 临时 owner，禁止多团队同时改。
2. 阶段 0 后改为“模块分段迁移”：每次只允许一个团队迁移一个能力簇并立刻委托到新模块。
3. 前端 `services/api.js` 与 `App.jsx` 视为共享层文件，改动必须有 Team-G 评审。
4. 每个任务合并前必须补充 `doc/refactor/test-records/` 记录。

## 6. 每日推进建议（给新 LLM）

1. 每天只推进 1~2 个小任务编号，不追求“大改一步到位”。
2. 先做结构迁移再做行为优化，确保回滚简单。
3. 合并后立即更新任务状态，避免多人重复拆同一块。
