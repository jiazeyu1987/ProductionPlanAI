# 契约冻结与变更流程

更新时间：2026-03-28  
适用范围：所有涉及接口（`/api`、`/v1`、`/internal/v1/internal`）的重构任务

## 1. 契约基线来源

1. 外部契约：`openapi/scheduling-v1.yaml`
2. 内部契约：`openapi/scheduling-internal-v1.yaml`
3. 运行时代码入口：
   1. `module/scheduler/controller/LegacyApiController.java`
   2. `module/integration/controller/ExternalContractController.java`
   3. `module/dispatch/controller/InternalContractController.java`

## 2. 冻结范围（Wave-1/Wave-2 必须保持）

1. 路由前缀不变：`/api/*`、`/v1/*`、`/internal/v1/internal/*`
2. 鉴权规则不变：`/v1/*` 与 `/internal/v1/*` 继续要求 Bearer token
3. 幂等语义不变：写接口继续要求 `request_id`
4. 分页结构不变：`request_id/page/page_size/total/items`
5. 错误模型不变：保持现有 `code/message/retryable/timestamp` 语义

## 3. 模块级契约 owner（用于变更审批）

1. Team-A：排产与版本相关接口（`schedule-versions*`、`/api/schedules*`、报表版本绑定）
2. Team-B：订单与报工接口（`order-pool*`、`/api/orders*`、`/api/reportings*`）
3. Team-C：主数据与日历接口（`masterdata/*`、`schedule-calendar/*`、`/v1/mes/process-routes`）
4. Team-D：调度指令/告警/重排/仿真接口（`dispatch-commands*`、`alerts*`、`replan-jobs*`、`simulation/*`）
5. Team-E：ERP/MES 集成接口（`/v1/erp/*`、`/v1/mes/*`、`integration/*`）
6. Team-G：鉴权、错误协议、request_id、CORS、共享 API 基础层

## 4. 变更分级

1. S0（无需契约评审）
   1. 仅内部重构，不改接口路径/字段/语义。
2. S1（兼容性变更，需模块评审）
   1. 仅新增可选字段或新增不影响旧调用方的接口。
3. S2（高风险变更，需跨团队评审）
   1. 修改必填字段、字段语义、错误码语义或默认行为。
4. S3（破坏性变更，默认禁止）
   1. 删除接口、重命名路径、删除字段。
   2. 若确需执行，必须先提供双写/双读兼容窗口与下线计划。

## 5. 契约变更标准流程

1. 发起变更单：说明变更级别（S0/S1/S2/S3）与影响模块。
2. 补充契约差异：更新 OpenAPI 或提供字段级 diff。
3. 评审通过后实施：先加兼容层，再切换调用方。
4. 执行契约测试：至少覆盖成功、鉴权失败、缺少 `request_id` 三类场景。
5. 更新文档：同步更新 `doc/refactor/` 与测试记录。
6. 验收通过后合并：未通过不得进入下一重构步。
