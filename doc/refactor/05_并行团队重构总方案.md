# 并行团队重构总方案

更新时间：2026-03-28  
适用范围：`code/frontend` + `code/backend` + `doc/refactor/*`

## 1. 背景与目标

项目将进入长期扩展阶段，当前代码已具备业务闭环，但存在“单体大文件、职责聚合、耦合偏高”的典型问题。  
本轮重构采用“**模块拆分 -> 团队并行 -> 分波次集成**”方式，目标如下：

1. 可扩展：新增业务不再修改核心大文件。
2. 可维护：模块边界清晰，团队可独立迭代。
3. 低耦合：跨模块通过稳定契约交互。
4. 可验证：每次重构后都可自动化验证，不靠人工猜测。

## 2. 总体策略

1. 先“边界重构”，后“实现重构”  
先确定模块职责和接口契约，再迁移代码实现。
2. 先“并行分治”，后“阶段集成”  
各团队在独立模块内小步改造，按固定节奏合并。
3. 先“测试护栏”，后“功能演进”  
每一小步必须通过对应测试，才能进入下一步。

## 3. 并行重构基本原则

1. 单模块独立负责制：一个模块由一个团队最终负责。
2. 契约优先：先冻结接口输入输出，再改内部实现。
3. 禁止跨模块直接改核心逻辑：跨模块变更必须走契约变更流程。
4. 一次只做一类风险：结构调整、逻辑变更、性能优化不混在同一小步。
5. 强制测试闸门：不通过测试不得进入下一次重构（详见 `07_分步重构与测试闸门.md`）。

## 4. 目标架构形态（重构后）

后端目标形态：

1. `api`：路由、鉴权、请求校验、协议转换。
2. `application`：用例编排（跨领域流程）。
3. `domain`：核心业务规则（订单、排程、主数据、执行、告警等）。
4. `infrastructure`：DB、ERP/MES、消息、文件导出。

前端目标形态：

1. `app-shell`：路由与布局。
2. `features/*`：按业务域拆分页面与状态逻辑。
3. `shared/*`：通用组件、hooks、api-client、工具库。
4. `lite-scheduler/*`：璞慧生产独立子域。

## 5. 重构输出物（每个模块都必须交付）

1. 模块职责说明（边界、输入、输出、依赖）。
2. 模块接口契约（request/response/error）。
3. 模块测试清单（单测、契约测、回归测）。
4. 迁移说明（旧路径 -> 新路径，兼容策略，回滚方式）。
