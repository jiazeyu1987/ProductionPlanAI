# 分步重构与测试闸门（强制执行）

更新时间：2026-03-28  
核心原则：**每重构一次，必须先测试验证通过，再进行下一次重构。**

## 1. 单步重构标准流程（必须按顺序）

1. 选定一个小步目标（单模块、单责任、单风险）。
2. 实施代码改动（不混入无关改造）。
3. 运行该模块最小测试集（单测/契约测/冒烟）。
4. 运行跨模块回归最小集（防止联动破坏）。
5. 通过后合并；失败必须先修复，不得继续下一步。

## 2. 测试分层

1. L1：模块单元测试（最快反馈）
2. L2：接口契约测试（接口结构与语义）
3. L3：业务回归测试（主流程）
4. L4：集成冒烟（前后端联动）

## 3. 当前项目建议测试命令

后端：

1. `cd code/backend && mvn test`

前端：

1. `cd code/frontend && npm test`
2. `cd code/frontend && npm run build`

全量验证：

1. `cd code && powershell -ExecutionPolicy Bypass -File .\\scripts\\run_verify.ps1`

## 4. 模块级必测清单（示例）

Team-A（排程与版本）：

1. 生成排产版本
2. 查看任务与算法解释
3. 发布与回滚
4. 差异比较

Team-B（订单与执行）：

1. 订单池读取与筛选
2. 订单 patch
3. 报工写入
4. 完工进度同步

Team-C（主数据日历）：

1. 主数据读取/保存
2. 路线 CRUD
3. 日历规则更新
4. 排产输入可用性

Team-D（指令告警）：

1. 指令创建与审批
2. 告警确认/关闭
3. 重排任务查询

Team-E（集成）：

1. ERP 刷新状态查询
2. 手工刷新
3. inbox/outbox 查询与重试

Team-F（璞慧生产）：

1. 本地场景读写
2. 按量/按天模式排程
3. 推进一天与手动报结束
4. 快照与导出

## 5. PR 合并闸门（必须全部满足）

1. 对应模块测试全部通过。
2. 影响到的跨模块冒烟通过。
3. 文档同步更新（至少更新模块变更与测试记录）。
4. 回滚方式明确（配置回滚/代码回滚/数据回滚）。

## 6. 禁止项

1. 测试失败继续叠加下一个重构任务。
2. 一个 PR 同时做大规模重命名 + 业务逻辑重写 + 性能改造。
3. 未更新文档直接合并结构性重构。

## 7. 测试记录模板（建议每次填写）

1. 重构步编号：
2. 变更模块：
3. 变更说明：
4. 执行测试：
5. 测试结果：
6. 风险与回滚：
7. 是否允许进入下一步：`YES/NO`

建议将记录统一存放在：`doc/refactor/test-records/`
