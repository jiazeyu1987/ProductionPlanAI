# 模块拆分与团队分工

更新时间：2026-03-28

## 1. 团队编组（建议）

1. Team-A：排程与版本中心（Schedule & Version）
2. Team-B：订单池与执行闭环（Order & Execution）
3. Team-C：主数据与日历（Masterdata & Calendar）
4. Team-D：调度指令与告警（Dispatch & Alert）
5. Team-E：ERP/MES 集成与运维通道（Integration）
6. Team-F：璞慧生产独立域（Lite Scheduler）
7. Team-G：平台基础与质量治理（Platform & QA）

## 2. 后端模块拆分矩阵

| 模块 | 当前主要代码 | 目标责任 | 负责团队 | 依赖边界 |
|---|---|---|---|---|
| ScheduleCore | `MvpStoreService` 中排产/版本逻辑、`SchedulerEngine` | 生成版本、差异、发布/回滚、算法解释 | Team-A | 仅依赖 Order/Masterdata/Integration 的只读端口 |
| OrderExecution | `MvpStoreService` 中订单池、报工、进度同步 | 订单管理、报工闭环、执行进度事实 | Team-B | 对外提供订单与进度查询端口 |
| MasterdataCalendar | `MvpStoreService` 中主数据/工艺/日历规则 | 工艺路线、资源能力、日历规则 | Team-C | 对排程提供配置快照端口 |
| DispatchAlert | 调度指令、审批、告警、重排触发 | 指令审批、告警状态、重排任务 | Team-D | 依赖 ScheduleCore 发布状态、OrderExecution 进度事实 |
| IntegrationHub | `ErpDataManager`、`ErpSqliteOrderLoader`、inbox/outbox | ERP/MES 数据接入、刷新、重试轨迹 | Team-E | 通过适配器向其他模块输出标准数据 |
| ApiContract | 各 Controller + `ApiSupport` + 鉴权过滤 | 路由协议、鉴权、request_id、统一错误协议 | Team-G | 不承载业务规则 |
| ObservabilityAudit | 审计、指标、日志规范 | 审计事件、质量指标、可观测规范 | Team-G | 与各业务模块事件集成 |

## 3. 前端模块拆分矩阵

| 模块 | 当前主要代码 | 目标责任 | 负责团队 | 依赖边界 |
|---|---|---|---|---|
| Schedule UI | `ScheduleBoard/Versions/Calendar` | 排程生成、版本比较、发布回滚界面 | Team-A | 只调用排程域 API Client |
| Order/Execution UI | `OrdersPool/ExecutionWip` | 订单池维护、报工录入、执行追踪 | Team-B | 只调用订单执行域 API Client |
| Masterdata UI | `MasterdataPage` | 工艺/资源/日历维护入口 | Team-C | 只调用主数据域 API Client |
| Dispatch/Alert UI | `DispatchCommands/Alerts/AuditLogs` | 指令审批、告警处置、审计查看 | Team-D | 只调用指令告警域 API Client |
| Integration/Ops UI | `OpsIntegration/Dashboard` 部分卡片 | 集成监控、消息重试、刷新状态 | Team-E | 只调用集成域 API Client |
| Lite Scheduler UI | `LiteSchedulerPage` + `liteSchedulerEngine` | 璞慧生产独立功能与演进 | Team-F | 与主系统通过导入导出/契约桥接 |
| Shell/Shared | `App.jsx`、`services/api.js`、通用组件样式 | 路由壳、全局组件、请求基础层 | Team-G | 为各 feature 提供底座能力 |

## 4. 目录目标（建议）

后端建议目录：

1. `backend/src/main/java/.../module/schedule/*`
2. `backend/src/main/java/.../module/orderexecution/*`
3. `backend/src/main/java/.../module/masterdata/*`
4. `backend/src/main/java/.../module/dispatchalert/*`
5. `backend/src/main/java/.../module/integration/*`
6. `backend/src/main/java/.../module/platform/*`

前端建议目录：

1. `frontend/src/features/schedule/*`
2. `frontend/src/features/order-execution/*`
3. `frontend/src/features/masterdata/*`
4. `frontend/src/features/dispatch-alert/*`
5. `frontend/src/features/integration/*`
6. `frontend/src/features/lite-scheduler/*`
7. `frontend/src/shared/*`

## 5. 团队协作规则

1. 一个 PR 只改一个模块主路径。
2. 跨模块改动必须附“契约变更说明”和“影响模块列表”。
3. 共享层（Team-G）改动需至少一个业务团队共同评审。
4. 每个模块维护自己的回归测试清单和冒烟脚本。
