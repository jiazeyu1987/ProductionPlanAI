# 当前业务流程梳理（重构基线）

更新时间：2026-03-28  
范围：`code/frontend`、`code/backend` 当前主实现（不含 `code/legacy-node`）

## 1. 目标与边界

本文档用于在重构前统一“当前系统实际怎么跑”的认知，重点覆盖：

1. 订单 -> 排产 -> 发布的主业务流程。
2. 执行报工、重排、告警、调度指令等核心闭环。
3. ERP/MES 集成与内部运营接口流程。

## 2. 当前系统模块视角

1. 前端（React）
2. 后端接口层（`/api`、`/v1`、`/internal/v1/internal`）
3. 后端业务核心（`MvpStoreService` + `SchedulerEngine`）
4. ERP 数据管理（`ErpDataManager` + `ErpSqliteOrderLoader`）

## 3. 业务主流程

### 3.1 订单进入与订单池管理

1. 前端订单池页读取 `/internal/v1/internal/order-pool`。
2. 后端按“本地订单 + ERP 生产单”合并返回订单池。
3. 运营在订单池可做 patch（锁单、冻结、优先级、交期等），写入后续排产输入。

### 3.2 排产生成（核心）

1. 前端排产看板触发 `/api/schedules/generate`（兼容接口）。
2. 后端在生成前同步订单完工进度，并构建排产输入快照。
3. `SchedulerEngine` 按策略（关键订单优先/最大产能优先/最小延期优先）生成：
   1. `tasks`
   2. `allocations`
   3. `unscheduled`（含阻塞原因）
4. 生成结果写入新版本（`DRAFT`），供看板/版本页查看。

### 3.3 版本发布与回滚

1. 前端版本页或看板调用 `/internal/v1/internal/schedule-versions/{versionNo}/publish`。
2. 后端将目标版本置为已发布，并处理前一发布版状态（被替代）。
3. 需要回退时调用 `/rollback`，后端将发布状态回滚并记审计。

### 3.4 执行报工与自动重排闭环

1. 前端执行页提交报工到 `/api/reportings`。
2. 后端记录报工并刷新订单完工量。
3. 满足触发条件时创建重排任务、产生告警，并写入集成消息轨迹（inbox/outbox）。
4. 前端可在告警页、集成页看到对应变化并做确认/关闭/重试。

### 3.5 调度指令审批闭环

1. 前端创建调度指令：`/internal/v1/internal/dispatch-commands`。
2. 审批人提交审批：`/dispatch-commands/{commandId}/approvals`。
3. 审批通过后后端应用指令到订单状态（如锁单/解锁/优先级调整等）。

### 3.6 主数据与排产规则维护

1. 前端主数据页读取：
   1. `/internal/v1/internal/masterdata/config`
   2. `/internal/v1/internal/schedule-calendar/rules`
   3. `/v1/mes/process-routes`
2. 用户保存配置或工艺路线 CRUD 后，后端更新当前运行态主数据。
3. 后续新生成排产版本立即使用新规则。

### 3.7 仿真流程

1. 前端仿真页支持批量/手动模式（当前默认手动）。
2. 后端提供：
   1. `run/reset`
   2. `manual/add-production-order`
   3. `manual/advance-day`
   4. `manual/reset`
3. 仿真会驱动订单、排产、报工、事件日志联动变化。

### 3.8 ERP/MES 集成流程

1. ERP 刷新有两种入口：
   1. 定时刷新（Scheduler）
   2. 手工刷新（内部接口）
2. 外部系统通过 `/v1/*` 获取 ERP/MES/报表数据。
3. 外部回写通过 `/v1/erp/schedule-results`、`/v1/erp/schedule-status`、`/v1/internal/*`。

## 4. 横切流程约束

1. 契约路由（`/v1`、`/internal/v1`）要求 Bearer 头。
2. 写接口要求 `request_id`（用于幂等与审计关联）。
3. 所有关键动作写审计日志，并在部分场景写入 inbox/outbox 轨迹。

## 5. 当前流程特征（重构视角）

1. 流程完整性较好，业务闭环基本具备。
2. 业务编排高度集中在单一服务，模块边界不清晰。
3. 多流程共享同一运行态对象，后续扩展时耦合风险高。
