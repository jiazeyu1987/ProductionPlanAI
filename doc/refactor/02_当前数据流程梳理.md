# 当前数据流程梳理（重构基线）

更新时间：2026-03-28  
范围：`code/frontend`、`code/backend` 运行态数据流

## 1. 总体数据流拓扑

1. 外部输入：前端操作、ERP 数据源（SQLite/API/CSV）、MES/ERP 回写请求。
2. 业务处理：后端接口层 -> `MvpStoreService` / `ErpDataManager` -> 排产引擎。
3. 输出对象：排产版本、报工、告警、审计、集成消息、报表文件（Excel）。

## 3. 主数据流明细

### 3.1 订单到排产数据流

1. 订单输入进入 `state.orders`（或由 ERP 订单并入订单池视图）。
2. 生成排产时先做“订单进度同步 + 物料约束快照”。
3. 组装排产输入副本 `scheduleState`，调用 `SchedulerEngine.generate`。
4. 输出版本写入 `state.schedules`，并生成任务、分配、未排原因与指标。

### 3.2 报工到告警/重排数据流

1. 报工写入 `state.reportings`。
2. 同步更新订单工序完工量与状态。
3. 达到阈值时触发：
   1. `state.replanJobs` 新任务
   2. `state.alerts` 新告警
   3. `state.integrationInbox/outbox` 消息轨迹

### 3.3 主数据配置到排产规则数据流

1. 主数据页提交配置（工序能力、线体拓扑、初始占用、物料可得量、日历规则）。
2. 后端更新 `state` 中对应结构体。
3. 后续排产生成直接使用更新后的主数据。

### 3.4 ERP 刷新到业务查询数据流

1. `ErpRefreshScheduler` 定时触发或手工触发刷新。
2. `ErpSqliteOrderLoader` 从 SQLite/API/CSV 拉取数据。
3. `ErpDataManager` 形成快照与缓存。
4. `/v1/erp/*`、订单池、物料可得性接口读取该快照参与业务结果计算。

## 4. 接口层数据协议特征

1. 契约接口统一分页响应结构：`request_id/page/page_size/total/items`。
2. 写操作强制 `request_id`，用于幂等与审计关联。
3. 存在双接口风格并行：
   1. `/api/*`（兼容风格）
   2. `/v1/*` 与 `/internal/v1/internal/*`（契约风格）

## 5. 前端侧数据流补充

1. 主业务页面主要通过后端接口读取/提交。
2. `LiteSchedulerPage` 使用前端本地轻量引擎，场景与快照保存在浏览器 `localStorage`。

## 6. 当前数据层现状（重构关注点）

1. 运行态核心数据以“内存状态对象”为中心，持久化能力较弱。
2. Flyway/DB 已具备表结构基础，但当前核心业务对象尚未形成完整 DB 驱动落库路径。
3. 幂等、状态、多流程数据共用单体内存结构，后续横向扩展与跨实例一致性风险较高。

## 7. 重构阶段建议的目标数据形态（方向）

1. 先将幂等账本、版本状态、审计、告警、指令审批迁移为持久化存储。
2. 再迁移订单进度事实、报工事实、重排任务与集成消息轨迹。
3. 最终形成“内存仅做缓存，数据库做事实源（Source of Truth）”的数据架构。
