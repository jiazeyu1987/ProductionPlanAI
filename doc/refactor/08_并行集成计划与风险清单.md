# 并行集成计划与风险清单

更新时间：2026-03-28

## 1. 集成节奏（建议 4 波次）

### Wave-1：边界冻结与骨架拆分

目标：

1. 冻结模块边界与接口契约。
2. 完成目录骨架与代码搬迁（逻辑不变）。

出口条件：

1. 编译通过。
2. 基础回归通过。
3. 文档完成更新。

### Wave-2：模块内部重构（并行）

目标：

1. 各团队在各自模块内拆分大函数/大页面。
2. 不改变对外契约，确保行为一致。

出口条件：

1. 模块测试通过。
2. 关键业务回归通过。

### Wave-3：跨模块能力打通

目标：

1. 规范跨模块调用（端口适配器/服务接口）。
2. 去除历史直连耦合。

出口条件：

1. 跨模块契约测试通过。
2. 全链路冒烟通过。

### Wave-4：整体验收与性能稳定

目标：

1. 全量回归 + 性能对比 + 故障演练。
2. 形成可持续迭代基线。

出口条件：

1. 质量门禁达标。
2. 发布检查清单全部通过。

## 2. 并行集成规则

1. 固定“集成窗口”合并（例如每周 2 次）。
2. 非窗口期仅允许紧急修复进入主分支。
3. 每次集成前锁定契约版本，避免接口漂移。
4. 集成失败时按模块回滚，不做整仓回滚。

## 3. 高风险点与控制措施

1. 风险：模块边界不清导致重复改造  
措施：先出模块责任清单并评审确认。

2. 风险：并行改造造成接口不兼容  
措施：契约版本化 + 契约测试必须在 PR 阶段通过。

3. 风险：大文件拆分后行为回归  
措施：拆分前后固定同一测试集，逐步比对。

4. 风险：多团队修改共享层冲突频繁  
措施：共享层由 Team-G 统一 owner，跨团队变更需联合评审。

5. 风险：重构过程中质量下滑  
措施：执行 `07` 文档强制闸门，不通过不得进入下一步。

## 4. 集成验收清单（每波次）

1. 后端 `mvn test` 通过。
2. 前端 `npm test` 与 `npm run build` 通过。
3. 一键验证脚本通过（`code/scripts/run_verify.ps1`）。
4. 核心业务链路人工抽检通过：
   1. 订单池
   2. 生成排产
   3. 发布/回滚
   4. 报工与告警
   5. 主数据保存
   6. 集成刷新/重试
5. 文档、迁移说明、回滚方案已更新。

## 5. 角色分工（集成阶段）

1. 模块团队：提交模块变更与模块测试报告。
2. Team-G（平台质量）：组织集成、执行闸门、发布结论。
3. 产品/业务代表：验证关键流程行为一致性。
