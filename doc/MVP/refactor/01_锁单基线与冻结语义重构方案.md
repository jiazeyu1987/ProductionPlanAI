# 01 锁单基线与冻结语义重构方案（v2）

## 1. 背景与问题
- 当前 `base_version_no` 仅写入版本元数据，未真正参与排程求解，导致“基于某版本重排”的业务语义不完整。
- 当前代码仅在 `autoReplan=true` 时排除锁单订单；`autoReplan=false` 不会自动排除。该行为与“锁单应被稳定保留”的业务预期存在偏差。
- 冻结订单在求解时被跳过，但未排原因默认落入 `CAPACITY_LIMIT`，造成策略性排除与资源瓶颈混淆。
- 以上问题影响调度安全、审计可解释性、以及版本对比可信度。

## 2. 重构目标与非目标
- 目标
- 让 `base_version_no` 成为真正输入，支持“基线继承 + 局部重排”。
- 锁单语义明确化：锁单任务在重排中默认保留，不参与优化漂移。
- 冻结语义明确化：冻结导致的未排单独归因，不再计入资源瓶颈。
- 保持现有接口向后兼容，新增字段以可选方式提供。
- 非目标
- 本方案不覆盖性能优化细节。
- 本方案不要求前端一次性完成全部可视化，仅定义后端契约与迁移路径。

## 3. 现状代码定位
- `MvpStoreService.generateSchedule`：创建版本并调用 `SchedulerEngine.generate`，但 `base_version_no` 未进入求解逻辑。
- `MvpStoreService.generateSchedule`：仅 `autoReplan=true` 时排除 `lockFlag=true` 的订单。
- `SchedulerEngine.generate`：冻结订单不分配，但未排原因没有策略类细分。
- `MvpStoreService.advanceManualOneDay`：生成并发布新版本，依赖上述语义一致性。

## 4. 统一原因码规范（跨文档共享）
- 新增标准原因码（建议统一在 `MvpDomain` 常量集中定义）
- `CAPACITY_MANPOWER`
- `CAPACITY_MACHINE`
- `MATERIAL_SHORTAGE`
- `DEPENDENCY_BLOCKED`
- `FROZEN_BY_POLICY`
- `LOCKED_PRESERVED`
- 兼容原因码
- 旧值 `CAPACITY_LIMIT` 保留读取能力。
- 旧值映射到 `CAPACITY_UNKNOWN`，禁止强制映射到 `CAPACITY_MANPOWER`。

## 5. 目标方案
- 基线继承
- 新增 `BaselineSnapshot`（只含“可继承计划事实”）：已分配任务、锁单任务、冻结任务、版本元信息。
- 禁止继承求解过程态（如 `producedQty` 临时变量），避免双计和状态污染。
- 锁单保留
- 锁单任务从 `BaselineSnapshot` 固化到新版本（默认保留时间窗和班次）。
- 若无可用基线且请求要求锁单保留，输出 `LOCKED_PRESERVED`/`LOCKED_NO_BASE` 说明并进入可审计未排集合。
- 冻结归因
- 冻结订单输出 `FROZEN_BY_POLICY`，不计入产能不足指标。
- 统计层将冻结与资源瓶颈分开聚合。

## 6. 分阶段实施
- 阶段 1：契约与模型
- 新增 `BaselineSnapshot` 与统一原因码常量。
- 扩展 `unscheduled` 结构：`reason_code`、`reason_detail`、`reason_source`。
- 阶段 2：求解入口改造
- `generateSchedule` 解析并加载 `base_version_no`，构造 `BaselineSnapshot`。
- `SchedulerEngine.generate` 增加 `baseline` 入参并执行“先固化、后求解”。
- 阶段 3：语义对齐
- 锁单、冻结进入统一归因通道。
- `advanceManualOneDay` 将原因分布写入 `daily_kpis` 与仿真事件详情。
- 阶段 4：发布与回滚
- 回滚只允许走 `rollbackSchedule` 业务流程。
- 禁止通过直接删除 `state.schedules` 做业务回滚。

## 7. 兼容与迁移
- 向后兼容
- 保持原 `reasons` 字段可读；新增 `reason_code` 并行输出。
- 历史版本缺失新字段时，前端与报表降级显示“未知原因”。
- 迁移策略
- 先在仿真链路启用（`manual/advance-day`）。
- 验证稳定后推广到常规生成与重排链路。

## 8. 风险与回滚
- 风险
- 基线固化范围定义过大，可能抑制优化空间。
- 原因码统计口径切换，可能造成短期 KPI 波动。
- 回滚
- 通过 feature flag 关闭“基线固化”与“新原因码输出”。
- 版本回退统一调用 `rollbackSchedule`，并保留审计链路。

## 9. 验收标准
- 语义正确性
- `autoReplan=true` 时锁单行为与基线保留策略一致。
- 冻结未排必须输出 `FROZEN_BY_POLICY`。
- 可解释性
- `unscheduled` 至少包含 `reason_code` 与 `reason_detail`。
- `daily_kpis` 可查看原因分布。
- 稳定性
- 现有 `SimulationPage.test` 与后端 API 回归测试全部通过。

## 10. 工时估算
- 设计与评审：1 人天。
- 后端实现与单测：2 人天。
- 集成验证与灰度：1 人天。
- 合计：4 人天。
