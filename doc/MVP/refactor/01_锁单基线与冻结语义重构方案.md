# 01 锁单基线与冻结语义重构方案（v3，状态更新）

> 更新日期：2026-03-24  
> 对齐代码：`MvpStoreService.generateSchedule`、`SchedulerEngine.generate`

## 1. 结论
本主题在当前代码中已完成核心落地，不再属于“未开始设计项”。

## 2. 已落地能力
1. `base_version_no` 已进入求解入口：
   - `generateSchedule` 会解析 `base_version_no` 并传入 `SchedulerEngine.generate(...)`。
2. 锁单保留语义已落地：
   - 锁单订单会进入 `lockedOrders`，用于基线保留；
   - 不再仅限 `autoReplan=true` 时才排除。
3. 冻结与策略性未排已独立归因：
   - `FROZEN_BY_POLICY`
   - `LOCKED_PRESERVED`
4. 任务/未排解释字段已透出到接口：
   - `task_status`
   - `dependency_status`
   - `last_block_reason`
   - `unscheduled_reason_code`

## 3. 原因码口径（当前实现）
1. `CAPACITY_MANPOWER`
2. `CAPACITY_MACHINE`
3. `CAPACITY_UNKNOWN`
4. `MATERIAL_SHORTAGE`
5. `DEPENDENCY_BLOCKED`
6. `FROZEN_BY_POLICY`
7. `LOCKED_PRESERVED`

## 4. 仍可优化项（未阻断当前MVP）
1. 基线继承 feature flag（当前为默认启用，未做动态开关）。
2. 更细粒度的“锁单无基线”专项码（当前以通用策略语义覆盖）。
3. KPI 看板对冻结/锁单影响的独立可视化深化。

## 5. 验收证据建议
1. 调度台与版本中心可看到锁单保留、冻结归因结果。
2. `MvpApiTest` 回归通过，且不再把策略性未排误归为单一容量原因。
