# 02 排程引擎可解释性与约束建模重构方案（v2）

## 1. 问题陈述
- 当前 `unscheduled` 原因过于粗糙，默认大量归入 `CAPACITY_LIMIT`，无法区分具体瓶颈。
- `calcAllowance` 对 FS/SS 依赖的表达偏保守，班次内前后工序释放关系可解释性不足。
- 版本任务接口缺少“为什么没排上”的标准字段，调度人员难以闭环定位。

## 2. 重构目标
- 建立统一、可扩展的约束原因码体系。
- 将依赖约束从“隐式规则”升级为“可观测状态”。
- 输出可用于前端、报表、审计的一致解释字段。
- 保持接口兼容，不破坏现有消费者。

## 3. 现状定位
- `SchedulerEngine.generate` 的 `unscheduled` 输出缺少结构化原因字段。
- `calcAllowance` 仅提供数值许可，缺少依赖阻塞的可解释上下文。
- `listScheduleTasks` 返回字段偏执行结果，不含约束解释。

## 4. 统一原因码与状态模型
- 原因码（与文档 01 一致）
- `CAPACITY_MANPOWER`
- `CAPACITY_MACHINE`
- `MATERIAL_SHORTAGE`
- `DEPENDENCY_BLOCKED`
- `FROZEN_BY_POLICY`
- `LOCKED_PRESERVED`
- 兼容码：`CAPACITY_LIMIT -> CAPACITY_UNKNOWN`
- 任务状态建议
- `READY`
- `PARTIALLY_ALLOCATED`
- `UNSCHEDULED`
- `PRESERVED_LOCKED`
- `SKIPPED_FROZEN`

## 5. 目标架构
- 解释对象
- 为每个未排任务输出 `reason_code`、`reason_detail`、`blocking_dimension`、`evidence`。
- 依赖对象
- 引入 `dependency_status`：`READY`、`WAIT_PREDECESSOR`、`BLOCKED_BY_PREDECESSOR`。
- 接口输出
- `schedule-versions/{version}/tasks` 增加 `dependency_status` 与 `last_block_reason`。
- `toScheduleMap.unscheduled` 增加结构化原因字段。

## 6. 实施步骤
- 步骤 1：结构定义
- 在领域模型增加解释字段，保证可序列化与可选输出。
- 步骤 2：求解内打标
- 在资源、物料、依赖、策略分支处分别打标准原因码。
- 禁止在未知情况下强制归因为人力不足。
- 步骤 3：依赖建模增强
- 保留 FS/SS 语义不变前提下，增加依赖阻塞快照输出。
- 如后续启用班次内释放策略，需以 feature flag 控制。
- 步骤 4：接口透出
- `listScheduleTasks` 与版本详情输出解释字段。
- 前端先以只读展示接入，不改变调度操作路径。

## 7. 测试策略
- 单元测试
- 覆盖每类原因码的触发分支。
- 覆盖 FS/SS 下 `dependency_status` 输出正确性。
- 集成测试
- 构造多约束冲突数据，校验原因优先级与字段完整性。
- 回归测试
- 历史 `CAPACITY_LIMIT` 仍可被正确读取。

## 8. 风险与回滚
- 风险
- 原因打标优先级不一致会导致解释抖动。
- 解释字段增多会带来 payload 增长。
- 回滚
- 通过 `showDetailedReasons` 开关关闭详细解释输出。
- 保留原 `reasons` 字段作为降级路径。

## 9. 里程碑
- M1（2 天）：完成原因码常量、模型字段、基础打标。
- M2（2 天）：完成依赖状态输出与接口透传。
- M3（1 天）：完成前端只读展示与回归验证。
- 合计：5 人天。
