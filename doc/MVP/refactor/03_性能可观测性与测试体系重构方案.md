# 03 性能可观测性与测试体系重构方案（v2）

## 1. 现状瓶颈
- `SchedulerEngine.generate` 主要是“班次 x 订单 x 任务”嵌套遍历，规模上来后响应抖动明显。
- `advanceManualOneDay` 串行执行多个阶段，缺少阶段耗时采集，慢点定位困难。
- 现有观测偏结果态，缺少过程态指标（候选任务量、未排原因分布、阶段耗时）。
- 测试偏功能正确性，缺少性能基线与回归门槛。

## 2. 指标体系
- 性能指标
- `schedule_generate_duration_ms`
- `schedule_generate_phase_duration_ms`
- `manual_advance_duration_ms`
- `manual_advance_phase_duration_ms`
- 质量指标
- `unscheduled_task_count`
- `unscheduled_reason_distribution`（原因码需与文档 01/02 一致）
- `schedule_completion_rate`
- `locked_or_frozen_impact_count`
- 可靠性指标
- `api_error_rate`
- `replan_failure_rate`
- `publish_rollback_count`

## 3. 性能重构路线
- 路线 A：索引化
- 构建 `orderNo/processCode/shiftId` 维度索引，减少重复扫描。
- 路线 B：候选集收敛
- 每班次先构建可排候选集合，再进入分配。
- 冻结、锁单任务提前分类并退出资源竞争路径。
- 路线 C：复杂度控制
- 任务完成快速剪枝。
- `calcAllowance` 结果缓存（按 task + shift）。
- 路线 D：推进链路分段
- `advanceManualOneDay` 分段计时并写入 summary 与事件。

## 4. 可观测性与告警
- 采集落点
- `SchedulerEngine.generate`：总耗时、阶段耗时、候选集规模、原因码分布。
- `MvpStoreService.advanceManualOneDay`：阶段耗时、总耗时、版本号关联。
- 告警建议
- `schedule_generate_duration_ms` P95 超阈值。
- `unscheduled_task_count` 或 `DEPENDENCY_BLOCKED` 异常突增。
- `manual_advance_duration_ms` 连续超阈值。
- 审计增强
- 新增 `perf_context` 字段并附带 `request_id/version_no/phase/duration_ms`。

## 5. 测试体系
- 单元测试
- 索引构建、剪枝逻辑、缓存命中、原因码统计。
- 集成测试
- `manual/advance-day` 全链路，验证阶段耗时字段与业务结果一致性。
- 基准测试
- 统一数据集：100 单、1000 单、5000 单。
- 输出：P50/P95、内存峰值、未排数量、原因分布。
- CI 策略
- 每日基准（夜间）+ PR 轻量基准（100/1000 单）。

## 6. 验收标准（优化后口径）
- 分档目标
- 100 单：P95 不劣化（<= 基线 + 5%）。
- 1000 单：P95 提升 >= 10%。
- 5000 单：P95 提升 >= 15%，且内存峰值不高于基线 + 10%。
- 质量红线
- 关键业务输出（版本号、报工、延期）与基线一致。
- 原因码分布可解释，禁止全部回落为单一原因。
- 可观测性红线
- 必须能在日志和 summary 中看到阶段耗时与原因分布。

## 7. 风险与回滚
- 风险
- 候选集收敛策略错误可能导致漏排。
- 过度打点可能反向增加开销。
- 回滚
- feature flag 可回退到旧求解路径。
- 打点可独立降级为仅总耗时。

## 8. 落地计划
- M1（2 天）：打点框架与指标落地，不改求解策略。
- M2（3 天）：索引化、候选集与剪枝改造。
- M3（2 天）：全链路联调、告警接入、基准回归。
- M4（1 天）：灰度复盘与阈值校准。
- 合计：8 人天。
