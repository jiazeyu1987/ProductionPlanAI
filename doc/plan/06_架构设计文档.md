# 架构设计文档（排产系统）

## 1. 架构目标
- 在不大幅增加复杂度的前提下，快速上线并稳定支持排产主流程。
- 支持后续扩展（更多规则、更多产线、更多系统对接）。

## 2. 总体架构（现行）
- 架构模式（MVP默认）：`模块化单体 + 数据库任务表异步`（`api`内含排产计算与任务调度）。
- 技术栈（MVP默认）：`Java 21 + Spring Boot + PostgreSQL + React`（`Redis`可选）。
- 技术栈（P1扩展）：`RabbitMQ + OR-Tools Worker + 独立sync/worker`。
- 口径来源：`doc/00_现行口径基线_2026-03-21.md`。
- 核心模块：
  1. 集成层（ERP/MES连接器）
  2. 订单中心
  3. 主数据中心（设备/工艺/日历）
  4. 排产引擎
  5. 资源约束服务（人机料可执行性）
  6. 调度台服务
  7. 执行反馈与重排服务
  8. 预警与审计服务

## 3. 关键组件说明

### 3.1 集成层
- 负责拉取 ERP、MES 数据并统一入库。
- 提供触发同步与夜间定时补偿同步。
- 提供增量同步、去重、重试、失败告警。

### 3.2 订单中心
- 维护订单待排池与状态机。
- 承接冻结、锁单、插单等控制动作。

### 3.3 主数据中心
- 管理产品-工艺-工序-设备映射。
- 管理班次日历、停机窗口。
- 管理员工技能矩阵与设备-工艺能力矩阵。

### 3.4 排产引擎
- 输入：订单池 + 工艺路线 + 产能 + 规则 + 人工指令。
- 输出：工序任务时序、订单预计开完工、瓶颈分析。
- 计算方式：先粗排后细排。

### 3.5 资源约束服务
- 校验工序任务可执行性（Man + Machine + Material）。
- 输出资源缺口原因（缺人/缺设备/缺料）。
- 支持冻结窗与局部/全局重排策略。

### 3.6 执行反馈与重排
- 持续消费 MES 报工。
- 与基线计划比较偏差，按规则触发重排。
- 生成新候选版本，避免直接覆盖已发布版本。
- 工序完工后生成WIP批次用于跨工序追踪。

## 4. 数据流（简化）
1. ERP销售订单入站 -> MRP链路数据（计划订单、投放状态）入站  
2. 计划订单投放后生产订单入站 -> 进入待排池（排产主体）  
3. MES工艺/设备入站 -> 主数据标准化  
4. 资源约束服务校验可执行性 -> 可执行任务集  
5. 排产引擎计算 -> 版本结果表  
6. 调度台人工调整 -> 版本发布  
7. 发布结果回写ERP + 通知  
8. MES报工到达 -> 偏差检测 -> 重排候选版本  
9. ERP采购入库事实入站 -> 即时库存事实更新（仅用于可视化/追溯，不参与P0排产求解）  

## 5. 存储设计（逻辑表）
- `order_pool`：订单待排池
- `process_route`：产品工艺路线
- `equipment_capacity`：设备产能
- `equipment_process_capability`：设备-工艺能力
- `employee_skill`：员工技能矩阵
- `shift_calendar`：班次日历
- `material_availability`：物料可用状态
- `schedule_version`：排产版本
- `schedule_task`：工序任务
- `wip_lot`：中间在制批次
- `dispatch_command`：人工调度指令
- `reporting_fact`：报工事实
- `alert_event`：预警事件
- `audit_log`：审计日志

## 6. 一致性与幂等
- 所有外部入站记录携带业务主键 + 更新时间。
- 采用“最后更新时间优先”做幂等更新。
- 发布动作必须原子化（版本状态切换与回写记录一致）。
- 报工按 `report_id` 幂等去重，WIP按 `wip_lot_id` 幂等创建。

## 7. 安全与审计
- 按功能分级授权（查看、调度、发布、管理）。
- 关键动作强审计（调序、锁单、发布、回滚）。
- 接口调用保留 request_id，支持链路追踪。

## 8. 可观测性
- 指标：同步延迟、计算耗时、发布成功率、预警命中率。
- 日志：入站失败、回写失败、重排触发原因。
- 告警：数据中断、队列积压、回写异常。

## 9. 部署约束（现行）
- 默认运行形态：脚本启动本地进程栈（`run_stack.ps1` / `code/scripts/run_stack.ps1`）。
- 默认端口：`frontend=5932`、`backend=5931`；前端默认调用后端 `http://localhost:5931`。
- 后端默认 profile：`local`（H2 内存库）；PostgreSQL 为联调/验收模式按需启用。
- P1扩展组件：`worker`、`sync`、`rabbitmq`。
- 关键要求：统一时区 `Asia/Shanghai`，生产库持久化，夜间补偿任务固定执行。
- 产能口径：`1班次=12小时`，每日开班数 `1` 或 `2`。
