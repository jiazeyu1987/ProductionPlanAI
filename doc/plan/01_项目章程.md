# 项目章程（排产系统）

## 1. 项目背景
- 当前排产模式为 `K3(ERP) + MES + Excel + 口头沟通` 并行，流程可运行但依赖人工经验。
- 周计划与临时调整分散在表格和沟通中，版本追溯、变更留痕和执行反馈效率偏低。
- 目标是建设“以订单交付为目标”的排产协同系统，形成可追踪、可复盘的闭环。

## 2. 项目目标
- 用系统承载排产主流程，减少对 Excel 和口头调度的依赖。
- 把现有规则（交期优先、加急插单、锁单、串并行工序）沉淀为可配置能力。
- 接入 MES 报工后自动更新进度并触发动态重排。
- 支持版本化发布与差异对比，确保计划变更可追溯。
- 统一班次口径与人机料约束，确保排产“可算且可执行”。

## 3. 项目范围（In Scope）
- 订单待排池与状态流转（待排、已排、执行中、完工、异常、冻结）。
- 排产规则引擎（优先级、插单、锁单、人工调序）。
- 有限产能排产（工序依赖、设备能力、班次日历）。
- 资源可执行性校验（人力技能、设备能力、物料可用）。
- 排产调度台（列表+甘特视图、拖拽调序、版本发布）。
- MES 报工接入与偏差预警、动态重排。
- 对 ERP 的计划结果回写与交付状态联动。

## 4. 非范围（Out of Scope）
- 库存可用量精算（安全库存/即时库存扣减）与补货优化。
- 高级成本优化与全局最优数学规划。
- 取代 ERP/MES 原生核心职能（如 ERP 财务、MES 采集控制）。

## 5. 关键业务边界
- `K3(ERP)`：订单、交期、审核/冻结状态、MRP关联、交付状态。
- `K3(MRP链路)`：销售订单 -> MRP运算生成计划订单 -> 计划订单投放（等同提交审核）-> 生产订单/采购申请。
- `K3(采购链路)`：采购申请 -> 采购订单 -> 收料通知/采购入库 -> 即时库存更新。
- `MES`：设备数据、工艺路线、工序数据、报工数据。
- `人工录入`：临时插单、优先级调整、异常决策（必须结构化留痕）。
- `排产对象边界`：排产以生产订单为对象；生产用料清单为生产订单的一对一附属数据，不作为排产驱动对象。
- `约束边界`：排产执行需同时满足人力（Man）、设备（Machine）、物料（Material）三类约束。
- `班次边界`：`1班次=12小时`，每日开班数为`1`或`2`（日班+夜班）。

## 6. 成功标准
- 系统内可独立完成周计划编排与发布。
- 报工到达后，系统可自动更新订单预计完成时间。
- 临时调整可追溯（谁在何时因何改动）。
- 延期风险可提前预警，并触发重排流程。

## 7. 风险与约束
- 上游数据口径不统一（订单状态、交期字段命名差异）。
- 人工口头调整如不入系统，会破坏排产一致性。
- 部分工序能力依赖经验估算，初期精度存在波动风险；按 `doc/plan/09_运行手册_Runbook.md` 执行监控与校准。
