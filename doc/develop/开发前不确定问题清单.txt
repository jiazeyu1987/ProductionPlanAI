已确认决策（2026-03-18）

A. 审计要求（对应原问题22）
1. 必审计操作：调序、发布、回滚、解冻、锁单/解锁、手工改优先级、手工改交期、同步任务重跑。
2. 审计字段：操作人、操作时间、对象ID、版本号、变更前、变更后、原因、来源IP/终端、request_id。
3. 控制要求：发布/回滚/解冻必须二次确认且原因必填；审计日志只追加不可改删。
4. 保留周期：审计日志至少保留3年，支持按订单/版本追溯导出。
5. 告警联动：回滚、批量解冻等高风险操作实时推送钉钉并抄送邮件。

B. 试点范围（对应原问题25）
1. 组织范围：首期1个车间，不跨工厂。
2. 业务范围：覆盖常规订单+加急订单的排产、调序、发布、回滚；首期不做库存优化。
3. 产品范围：20-30个高频SKU，覆盖约70%-80%日常订单量。
4. 系统范围：仅对接K3+MES；口头规则必须转为系统参数。
5. 排除范围：外协复杂场景、未固化工艺新品、跨车间联动排产。
6. 试点通过标准：
   - 排产计算P95 <= 5分钟
   - 同步成功率 >= 99%
   - 发布成功率 >= 99.5%
   - 连续2周无P1事故

C. 上线策略（对应原问题26）
1. 阶段1（双轨）：系统出计划但不主导执行，现场继续按Excel执行，每日对比偏差并修正规则。
2. 阶段2（受控切换）：系统发布为主，Excel兜底，仅白名单人员可发布/回滚。
3. 阶段3（正式切换）：系统成为唯一发布入口，Excel仅作应急备份。
4. 回滚策略：保留最近3个已发布版本；触发条件包括同步中断>30分钟、连续3次计算超时、连续2次发布失败；触发后一键回滚到上个稳定版本。
5. 运行策略：夜间固定执行增量对账+补偿同步，次日上班前输出差异报告。

补充：本轮同时明确的基础约束（可关闭原问题18-21、23-24）
1. 部署方式：本地机房单机部署（Docker容器化）。
2. ERP/MES对接方式：通过API对接。
3. 数据同步方式：触发同步 + 每晚定时补偿同步。
4. 权限模型：调序、发布、回滚、解冻等权限可配置。
5. 性能目标：单次排产计算可接受时长5分钟以内。
6. 告警通道：钉钉、邮件、系统内消息。

D. 技术栈选型讨论记录（对应原问题17）
1. 方案1（Java稳健型）
   - 技术栈：Java 21/25 + Spring Boot + PostgreSQL + Redis + RabbitMQ + OR-Tools Worker + React
   - 前端细化：React + TypeScript + Vite + Ant Design（或MUI）+ Redux Toolkit（或Zustand）+ React Query
   - 说明：前端从Vue改React可行且常见，后端架构基本不变。
   - 优点：企业生态成熟、事务与并发能力强、长期维护稳定。
   - 缺点：工程复杂度与上线速度通常不及轻量方案。

2. 方案2（.NET一体化）
   - 技术栈：.NET 10 + ASP.NET Core + EF Core + SQL Server（或PostgreSQL）+ Redis + RabbitMQ + OR-Tools(C#) + Vue3/React
   - 优点：工程化完整、交付效率高、企业集成能力强。
   - 缺点：若团队以Java/Python为主，迁移学习成本较高。

3. 方案3（Python快速迭代型）
   - 技术栈：FastAPI + PostgreSQL + Redis/RabbitMQ + OR-Tools(Python) + Vue3/React
   - 优点：规则与算法迭代速度快，PoC效率高。
   - 缺点：大型长期治理、复杂工程约束通常弱于Java/.NET。

4. 当前讨论结论
   - “方案1 + React”明确可行，可采用Docker部署。
   - 当前对话偏向方案1落地（Java后端不变，前端改React）。
   - 若后续无新变更，可将原问题17标记为“已确认：方案1（Java+React）”。

E. 部署与运行补充（已在对话确认）
1. 单机Docker推荐组件：
   - nginx（统一入口）
   - frontend（React静态站点）
   - api（Spring Boot主服务）
   - worker（排产计算服务）
   - sync（同步任务服务：触发+夜间定时）
   - postgres、redis、rabbitmq（基础组件，均需持久化）
2. 同步机制：
   - 触发同步：ERP/MES事件触发后异步入队处理。
   - 夜间定时：增量对账+补偿同步，修复漏数与状态漂移。
3. 关键技术要求：
   - 幂等：business_key + update_time 去重。
   - 重试与补偿：指数退避 + 死信队列 + 定时补偿。
   - 时区统一：全部容器Asia/Shanghai。
   - 生产保障：数据库备份、MQ持久化、关键日志与审计可追溯。

G. 最新通话确认（2026-03-18，MRP/采购/生产链路）
1. 销售订单通过MRP运算生成计划订单；计划订单投放（等同审核提交）后，触发生产订单与采购申请。
2. 生产订单生成时同步创建生产用料清单，二者为一一对应关系。
3. 排产依据为生产订单，不是生产用料清单。
4. 生产订单按系统预设工艺流程进行排产。
5. 采购申请下推采购订单；采购订单入库后，联动收料通知单与采购入库单，并更新即时库存。
6. 对当前项目边界的影响：排产MVP仍按既定原则“不做库存优化”，但可保留库存事实数据接口用于后续扩展。
