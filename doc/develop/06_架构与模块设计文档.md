# 架构与模块设计文档（开发版）

## 1. 架构选型
- 模式（MVP默认）：模块化单体（`api`内含排产引擎）+ 数据库任务表异步处理。
- 原因：使用人数不多、并发压力有限，优先保证上线速度、排障简单、运维成本低。
- 技术栈（MVP）：`Java 21 + Spring Boot + PostgreSQL + React`；`Redis`可选（缓存/限流），`RabbitMQ/独立Worker`放入P1扩展。
- 口径来源：`doc/00_现行口径基线_2026-03-21.md`。

## 2. 模块划分
1. `integration`：ERP/MES 数据接入与回写  
2. `order`：订单池与状态机  
3. `masterdata`：工艺、设备、班次日历、技能矩阵  
4. `resource`：人机料可执行性校验与瓶颈识别  
5. `scheduler`：排产计算与重排  
6. `dispatch`：人工调度与版本发布  
7. `execution`：报工消费、WIP批次与进度更新  
8. `alert`：预警规则与通知  
9. `audit`：操作日志与追溯  
10. `sla`：时效指标计算与SLA告警（评审、MRP、外围、交付）
11. MVP阶段上述模块同进程部署（逻辑分层，不做物理拆服务）。

## 3. 核心流程时序
1. `integration`拉入销售订单与计划订单链路  
2. 计划订单投放后生成生产订单 -> `order`入池  
3. `masterdata`准备工艺、设备、技能、班次、物料可用信息  
4. `resource`执行人机料约束校验 -> 形成可执行任务集  
5. `scheduler`读取生产订单+主数据 -> 生成草稿版本  
6. `dispatch`人工调整 -> 发布版本  
7. `integration`回写ERP结果  
8. `execution`消费MES报工 -> 更新任务进度与WIP批次  
9. 偏差达到阈值 -> `scheduler`重排 -> 新候选版本  

## 4. 任务处理模型
- 同步任务：数据拉取、回写
- 异步任务：批量排产、重排计算、预警计算、WIP事件处理
- 任务队列（MVP）：使用 `job_task` 数据表 + `@Scheduled` 轮询执行，保证重试、失败补偿、可观测
- 同步策略：触发同步优先，夜间补偿任务做漏数修复与对账
- 扩展策略：当任务量持续增长再拆分独立 `worker/sync` 进程，不提前复杂化。

## 5. 一致性策略
- 入站幂等：业务主键+更新时间去重。
- 发布幂等：同 `version_no` 不重复发布。
- 回写幂等：同 `request_id` 重复提交仅生效一次。
- 执行幂等：同 `report_id` 报工仅落账一次；同 `wip_lot_id` 仅创建一次。

## 6. 可观测性
- 指标：同步延迟、排产耗时、发布成功率、回写成功率、重排次数、瓶颈负荷率。
- 日志：接口错误、重排触发原因、人工干预记录、WIP转序事件。
- 告警：任务表堆积、数据中断、回写失败率超阈值、关键资源不可用。

## 7. 边界与职责
- ERP只负责订单与状态事实，不负责排产求解。
- MES只负责设备/工艺/报工事实，不负责计划版本管理。
- 排产系统负责规则求解、版本发布、偏差重排与预警。
- 排产对象为ERP生产订单；生产用料清单仅作追溯，不作为排产引擎输入主体。
- 班次口径固定：`1班次=12小时`，每日开班数为`1`或`2`。

## 8. 安全与权限
- 权限角色：查看、调度、发布、管理员。
- 关键操作（发布、回滚、锁单）强制审计。
- 调序、发布、回滚、解冻权限点可配置。

## 9. 部署约束
- 默认运行形态：脚本启动本地进程栈（`run_stack.ps1` / `code/scripts/run_stack.ps1`）。
- 默认端口：`frontend=5932`、`backend=5931`；前端默认调用后端 `http://localhost:5931`。
- 后端默认 profile：`local`（H2 内存库）；PostgreSQL 作为联调/验收模式按需启用。
- 非MVP必需组件：`worker`、`sync`、`rabbitmq`（仅在P1扩展时启用）。
- 运行约束：单次排产计算目标 P95 <= 5 分钟，统一时区 `Asia/Shanghai`。

## 10. 后端优化建议与实施优先级（新增）

### 10.1 优化项清单
1. 增量重排替代全量重排：订单变更后仅重算受影响工序子图，减少重排耗时与扰动范围。  
2. 产能预聚合：预计算 `工艺 x 班次 x 人机料可用量` 快照，排程阶段直接读取，降低在线计算压力。  
3. 两阶段求解：先做硬约束可行性（人/机/料/班次/工艺），再做软目标优化（交期、切换成本、负荷均衡）。  
4. 结果可解释：对“不可排/延期/冲突”输出结构化原因（缺设备、缺技能工、缺料、冻结窗命中）。  
5. 长耗时任务异步化：排程、回滚、批量调整统一进入任务队列，前端通过任务ID查询进度。  
6. 并发控制细化：按 `订单/工序/班次` 粒度做乐观锁或细粒度锁，避免整表锁与互相阻塞。  
7. 集成稳态增强：入站暂存 + 幂等账本 + Outbox + 退避重试 + 死信队列，提升 ERP/MES 抖动下稳定性。  
8. 热冷分层：近90天热数据高性能存储，历史归档按月分区，保证查询与写入性能稳定。  
9. 观测链路打通：统一 `request_id` 贯穿 API 日志、事件日志、审计日志、告警上下文。  
10. 回放式回归：基于真实历史订单做排程回放，对比准交率、重排次数、求解耗时等KPI后再发布。  

### 10.2 推荐实施顺序
1. `P0-MVP必做`：第1项增量重排、第4项结果可解释、第5项异步化（基于任务表）。  
2. `P0-MVP建议`：第2项产能预聚合、第6项并发控制、第9项观测链路。  
3. `P1-扩展再做`：第3项两阶段求解、第7项集成稳态增强、第8项热冷分层、第10项回放式回归。  

## 11. MVP最小交付范围（快速上线）
1. 单服务完成订单入池、主数据管理、排程计算、版本发布、执行反馈、审计追溯。  
2. 排程任务与同步补偿统一走数据库任务表，不引入独立消息中间件。  
3. 对外仍保持现有接口契约，内部实现优先“简单可维护”，避免过度抽象。  
4. 首版只保证核心链路：`待排池 -> 调度台 -> 发布 -> 报工反馈 -> 预警`，其余高级优化按P1迭代。
