# 内部接口与事件契约（P0，现行）
> 文档属性：现行开发规范。  
> 生效日期：2026-03-22。  
> 冲突处理：以 `doc/00_现行口径基线_2026-03-21.md` 为准。

## 1. 目的与范围
1. 统一前端页面、排产服务、执行服务、预警服务之间的内部接口契约。
2. 统一内部事件消息体，确保“写库、回写、通知、重排”链路可追溯。
3. 本文不替代 ERP/MES 外部接口；外部接口仍以 `scheduling-v1` 为准。

## 2. 协议与通用约束
1. 协议：`HTTPS + JSON`。
2. 鉴权：JWT + RBAC（二次鉴权）。
3. 幂等：所有写接口强制 `request_id`。
4. 分页：`page/page_size`，默认 `200`，最大 `1000`。
5. 时区：统一 `Asia/Shanghai`，时间字段均为 ISO8601。
6. 错误体：统一复用 `ErrorResponse` 结构。
7. 路径约定：本文 3.x/4.x 使用“内部契约相对路径”；实际调用完整路径为 `/internal/v1` + 相对路径（例如 `/internal/v1/internal/order-pool`）。

## 3. 内部 REST 接口（P0最小集）

### 3.1 待排池与调度数据
1. `GET /internal/order-pool`：查询待排订单池。
2. `GET /internal/schedule-versions`：查询版本列表。
3. `GET /internal/schedule-versions/{version_no}/tasks`：查询版本任务明细。
4. `GET /internal/schedule-versions/{version_no}/diff`：查询版本差异摘要。

### 3.2 调度指令与审批
1. `POST /internal/dispatch-commands`：创建调度指令。
2. `POST /internal/dispatch-commands/{command_id}/approvals`：审批指令。
3. `GET /internal/dispatch-commands`：查询指令与审批状态。

### 3.3 版本发布与回滚
1. `POST /internal/schedule-versions/{version_no}/publish`：发布版本。
2. `POST /internal/schedule-versions/{version_no}/rollback`：回滚到目标版本。

### 3.4 预警与重排
1. `GET /internal/alerts`：查询预警事件。
2. `POST /internal/alerts/{alert_id}/ack`：确认预警。
3. `POST /internal/alerts/{alert_id}/close`：关闭预警。
4. `POST /internal/replan-jobs`：触发重排任务。
5. `GET /internal/replan-jobs/{job_no}`：查询重排任务状态。

### 3.5 审计与同步运维
1. `GET /internal/audit-logs`：查询审计日志。
2. `GET /internal/integration/inbox`：查询入站消息。
3. `GET /internal/integration/outbox`：查询出站消息。
4. `POST /internal/integration/outbox/{message_id}/retry`：手动重试投递。

### 3.6 仿真
1. `GET /internal/simulation/state`：查询仿真状态摘要。
2. `GET /internal/simulation/events`：查询仿真事件明细。
3. `POST /internal/simulation/reset`：重置仿真状态。
4. `POST /internal/simulation/run`：按参数运行仿真。
5. `POST /internal/simulation/manual/add-production-order`：手动模拟新增生产订单（主系统生效）。
6. `POST /internal/simulation/manual/advance-day`：手动模拟推进一天（主系统排产+报工推进）。
7. `POST /internal/simulation/manual/reset`：恢复到手动模拟前快照。

## 5. 事件契约（MVP数据库事件表 / P1 RabbitMQ Topic）
1. MVP阶段默认采用“事件落表 + 任务轮询分发”，不强依赖 RabbitMQ。
2. P1扩展可切换到 RabbitMQ Topic；事件名与消息体字段保持不变。
3. 下列事件名为统一契约：
4. `scheduling.reporting.ingested`：报工入库完成。
5. `scheduling.wip.event.created`：WIP 流转事件创建。
6. `scheduling.replan.triggered`：重排任务触发。
7. `scheduling.version.published`：版本发布完成。
8. `scheduling.version.rollbacked`：版本回滚完成。
9. `scheduling.alert.created`：预警事件生成。
10. `scheduling.alert.closed`：预警事件关闭。
11. `scheduling.outbox.delivery.failed`：出站投递失败。

事件公共字段：
1. `event_id`（全局唯一）
2. `event_type`
3. `occur_time`
4. `request_id`
5. `trace_id`
6. `producer`
7. `payload`

## 6. 幂等与补偿
1. 写接口幂等键：`request_id + biz_key + action`。
2. 消费幂等落账：`idempotency_ledger` 记录成功结果哈希。
3. 投递失败进入 `integration_outbox` 重试队列。
4. 超过重试上限进入人工处理队列并触发告警。

## 7. 审计要求
1. 所有写操作必须记录 `operator/request_id/reason/client_ip`。
2. 审批/发布/回滚必须记录前后状态快照。
3. 内部接口操作可按 `request_id` 回溯至消息投递链路。
