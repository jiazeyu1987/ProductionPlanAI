# 数据库设计文档（排产系统）

## 0. 实施口径（2026-03-22）
1. 本文同时包含“逻辑数据模型”和“当前MVP物理落地表”两层描述。  
2. 逻辑数据模型用于业务口径、字段约束和后续扩展，不代表当前版本已全部物理建表。  
3. 当前MVP物理落地（Flyway）以最小闭环为主，已落地表如下：  
   - `job_task`、`idempotency_ledger`、`audit_log`、`replan_job`、`alert_event`  
   - `dispatch_command_approval`、`schedule_publish_log`、`integration_inbox`、`integration_outbox`、`notification_delivery_log`  
   - `mvp_dict_cn`  
4. ERP销售/生产订单与周/月计划报表数据当前由服务内存态聚合，并可从外部 SQLite（`mvp.erp.sqlite-path`）读取；对外通过接口提供主业务事实层与原始明细层。

## 1. 设计原则
- 业务主键明确，支持幂等更新。
- 状态变更可追溯。
- 排产结果版本化存储。
- 班次口径统一（`1班次=12小时`）。
- 支持人机料约束的可执行性校验。

## 2.15 表作用总览（现有表）
1. `order_pool`：订单待排池，承接可排产准入状态、交期、冻结等核心排产入口信息。  
2. `plan_order`：计划订单事实，用于追溯 MRP 投放链路与来源关系。  
3. `process_route`：产品工艺路线与工序参数，提供细排、串并行、产能计算输入。  
4. `equipment_capacity`：设备可用状态与班次产能基础信息。  
5. `equipment_process_capability`：设备-工艺能力矩阵，判断设备是否可执行某工艺。  
6. `employee_skill`：员工-工艺技能矩阵，支撑人力可执行性校验。  
7. `shift_calendar`：班次日历，定义日班/夜班开班与时段边界。  
8. `material_availability`：物料齐套与可用时间事实，支撑缺料拦截。  
9. `schedule_version`：排产版本头，管理草稿/发布/回滚状态。  
10. `schedule_task`：工序级排产任务明细，存储计划开始/结束/数量等结果。  
11. `reporting_fact`：MES 报工事实，驱动进度更新与重排触发。  
12. `wip_lot`：中间在制批次（工艺中间产品）主表。  
13. `dispatch_command`：人工调度指令（插单/锁单/解冻/调序）记录。  
14. `audit_log`：关键操作审计日志，满足可追溯要求。  

## 2.16 关系约束（完善版）

### 2.16.1 订单主链路关系
1. `sales_order_line_fact(sales_order_no, line_no)` 1:N `production_order_fact(source_sales_order_no, source_line_no)`（逻辑外键）。
2. `plan_order(plan_order_no)` 1:N `production_order_fact(source_plan_order_no)`（逻辑外键）。
3. `production_order_fact(production_order_no)` 1:1 `order_pool(order_no, order_type='production')`（逻辑映射）。
4. `schedule_control_fact(order_no, order_type)` 1:N `order_pool(order_no, order_type)`（逻辑外键，控制粒度高于订单行粒度）。
5. `mrp_result_link(order_no, order_type)` N:1 `order_pool(order_no, order_type)`（逻辑外键）。
6. `delivery_progress(order_no, order_type)` 1:N `order_pool(order_no, order_type)`（逻辑外键，进度粒度高于订单行粒度）。
7. `production_material_list_item(production_order_no)` N:1 `production_order_fact(production_order_no)`（逻辑外键）。

### 2.16.2 主数据与资源关系
1. `product(product_code)` 1:N `product_route_header(product_code)`。
2. `product_route_header(route_no)` 1:N `process_route(route_no)`。

6. `process(process_code)` 1:N `route_step_dependency(from_process_code/to_process_code)`。
7. `process(process_code)` 1:N `process_resource_requirement(process_code)`。
8. `equipment_capacity(equipment_code)` 1:N `equipment_process_capability(equipment_code)`。
9. `process(process_code)` 1:N `equipment_process_capability(process_code)`。
10. `employee(employee_id)` 1:N `employee_skill(employee_id)`。
11. `process(process_code)` 1:N `employee_skill(process_code)`。
12. `organization_unit(unit_code)` 1:N `equipment_capacity(workshop_code/line_code)`（逻辑外键）。
13. `organization_unit(unit_code)` 1:N `shift_calendar(workshop_code)`（逻辑外键）。
14. `organization_unit(unit_code)` 1:N `employee(team_code)`（逻辑外键）。
15. `organization_unit(parent_unit_code)` 1:N `organization_unit(unit_code)`（自关联）。

### 2.16.3 排产版本与任务关系
1. `rule_param_version(rule_version_no)` 1:N `schedule_version(rule_version_no)`。
2. `schedule_version(version_no)` 1:N `schedule_task(version_no)`。
3. `order_pool(order_no, order_type)` 1:N `schedule_task(order_no, order_type)`（逻辑外键）。
4. `schedule_task(id)` 1:N `schedule_task_resource(task_id)`。
5. `schedule_task(id)` 1:N `schedule_task_change_log(task_id)`。
6. `schedule_version(version_no)` 1:N `order_priority_factor(version_no)`。
7. `order_pool(order_no, order_type)` 1:N `order_priority_factor(order_no, order_type)`（逻辑外键）。
8. `schedule_version(version_no)` 1:N `replan_job(base_version_no/result_version_no)`（逻辑外键）。
9. `schedule_version(version_no)` 1:N `schedule_publish_log(from_version_no/to_version_no)`（逻辑外键）。
10. `dispatch_command(command_id)` 1:N `dispatch_command_approval(command_id)`。
11. `dispatch_command(target_order_no, target_order_type)` 逻辑关联 `order_pool(order_no, order_type)` 的同键订单行集合（订单头粒度）。

### 2.16.4 执行、WIP、预警与通知关系
1. `order_pool(order_no, order_type)` 1:N `reporting_fact(order_no, order_type)`（逻辑外键）。
2. `reporting_fact(report_id)` 1:N `wip_lot(source_report_id)`（逻辑外键）。
3. `wip_lot(wip_lot_id)` 1:N `wip_lot_event(wip_lot_id)`。
4. `reporting_fact(report_id)` 1:N `wip_lot_event(source_report_id)`（逻辑外键）。
5. `equipment_capacity(equipment_code)` 1:N `equipment_downtime_event(equipment_code)`。
6. `employee(employee_id)` 1:N `employee_shift_availability(employee_id)`。
7. `schedule_version(version_no)` 1:N `alert_event(version_no)`（逻辑外键）。
8. `order_pool(order_no, order_type)` 1:N `alert_event(order_no, order_type)`（逻辑外键）。
9. `alert_event(alert_id)` 1:N `notification_delivery_log(alert_id)`。

### 2.16.5 集成、幂等与同步关系
1. `integration_inbox(message_id)` 与业务事实表之间为“解析写入”关系（应用层映射，不做物理外键）。
2. `integration_outbox(message_id)` 与外部回写/通知为“投递结果”关系（应用层映射，不做物理外键）。
3. `idempotency_ledger(request_id, biz_key, action)` 为幂等唯一键，覆盖入站处理、发布、回写动作。
4. `sync_checkpoint(source_system, entity_name)` 为增量游标唯一键，覆盖触发同步与补偿同步。

### 2.16.7 逻辑外键（跨系统/多态键）
1. 所有 `order_no + order_type` 关联（`schedule_task/reporting_fact/schedule_control_fact/mrp_result_link/delivery_progress/alert_event/dispatch_command`）按逻辑外键处理。
2. `production_order_fact` 与 `order_pool` 的 1:1 映射按规则校验处理（`order_type='production'`）。
3. `wip_lot.source_report_id`、`wip_lot_event.source_report_id` 因上游补偿重放场景，按逻辑外键处理。
4. `schedule_publish_log(from_version_no/to_version_no)` 与 `schedule_version` 按逻辑外键处理（支持空值与失败日志）。
5. `production_material_list_item.production_order_no` 与 `production_order_fact.production_order_no` 按逻辑外键处理（处理到达乱序）。

说明：逻辑外键必须配套“定时一致性校验任务 + 异常修复流程”。

## 4. 数据保留策略（执行）
- 报工事实：长期保留。
- WIP批次：至少保留2年明细，历史可归档。
- WIP流转事件：至少保留2年明细，历史可归档。
- 排产任务版本：保留最近20个完整版本，历史版本可归档。
- 发布/回滚日志：至少保留3年。
- 审计日志：按合规要求保留。
- 设备停机事件：至少保留2年，支持可靠性分析与重排追溯。
- 员工班次可用性：至少保留1年，用于排班争议追溯。
- 任务级变更日志：至少保留1年（增强项启用时）。
- 预警事件：至少保留1年，重大告警建议保留2年。
- 规则参数版本：长期保留（作为排产结果解释依据）。
- 通知投递日志：至少保留1年（增强项启用时）。
- 销售订单行/生产订单/准入控制/MRP链路/交付进度事实：至少保留2年。
- 生产用料清单明细：至少保留2年（用于追溯与稽核）。
- 订单评分快照：至少保留1年，且建议保留最近20个发布版本对应快照。
- 入站/出站消息与幂等账本：至少保留1年，支持重放与稽核。
- 逻辑删除数据：默认不物理删，按归档策略转冷存。
