# 前端页面与交互规格（P0，现行）
> 文档属性：现行开发规范。  
> 生效日期：2026-03-22。  
> 冲突处理：以 `doc/00_现行口径基线_2026-03-21.md` 为准。

## 1. 目标与范围
1. 将 P0 功能需求拆解为可直接开发的页面、路由、组件和交互规则。
2. 统一“页面按钮权限、操作审批、接口调用、审计留痕”的前后端实现边界。
3. 本文只覆盖 P0，P1/P2 增强能力不在本页面规格内。

## 3. 页面规格
实现状态说明（避免口径歧义）：
1. 本章节同时包含“当前MVP已实现最小能力”和“后续增强目标能力”。
2. 2026-03-22 当前实现以 `code/frontend/src/pages` 为准，重点已落地页面：`/orders/pool`、`/schedule/board`、`/schedule/versions`、`/dispatch/commands`、`/execution/wip`、`/alerts`、`/audit/logs`、`/masterdata`、`/ops/integration`、`/reports/plans`、`/simulation`、`/guide`。
3. 尚未实现的增强能力（如甘特拖拽、复杂图表、高级筛选持久化）按“后续规划”处理，不作为当前MVP验收阻断项。

### 3.1 经营看板 `/dashboard`
当前MVP实现状态：已实现基础指标卡与“今日必须处理”列表；高级筛选、趋势图与指标下钻为后续增强项。
1. 顶部筛选：日期范围、产品线、车间、订单类型。
2. 核心指标卡：准交率、延期订单数、计划达成率、重排次数、ERP 回写成功率、排产 P95。
3. 增加“今日必须处理”区：仅展示最紧急订单、阻断原因和建议动作。
4. 趋势图：近 7/30 天延期风险、不可执行任务占比、关键工艺人力覆盖率。
5. 明细穿透：点击任意指标卡必须下钻到明细列表并自动带筛选条件。
6. 数据刷新：默认 60 秒自动刷新，支持手动刷新。

### 3.1A 调度驾驶舱（单页作战）
1. `Dashboard` 提供“单页作战模式”，将常看数据、异常队列、任务列表、常用操作集中在一个页面。
2. 布局要求：
   - 顶部：核心指标卡 + 今日必须处理。
   - 左侧：异常优先队列（延期风险/缺料/设备冲突/人力不足/冻结窗冲突）。
   - 中部：清晰数据列表（支持固定列、快速筛选、批量勾选）。
   - 右侧：常用操作快捷栏（插单、锁单/解锁、冻结/解冻、调优先级、触发重排、发布版本）。
3. “原始信息查看”能力必须内置抽屉：展示来源系统、原始字段、更新时间、`request_id/message_id`、变更记录。
4. 目标：80% 日常调度操作在单页内完成，避免跨页面高频跳转。

### 3.2 待排池 `/orders/pool`
当前MVP实现状态：已实现列表、刷新、行内快捷操作（锁单/解锁/提优先级）；高级过滤与批量冻结审批为后续增强项。
1. 列表列：订单号、类型、产品、数量、交期字段、加急、评审/冻结/关闭状态、可排产标识。
2. 过滤条件：可排产状态、冻结状态、加急标识、交期区间、产品编码、客户。
3. 行内操作：锁单、解锁、标记人工权重、发起插单请求。
4. 批量操作：批量锁单、批量顺延、批量冻结（触发审批流）。
5. 校验规则：`review_passed_flag=1 && frozen_flag=0 && close_flag=0 && schedulable_flag=1` 才允许进入排程请求。

### 3.3 调度台 `/schedule/board`
当前MVP实现状态：已实现版本选择、任务列表与生成草稿版本；甘特拖拽、冲突可视化与撤销/重做为后续增强项。
1. 视图模式：甘特图与列表双视图可切换，同步同一筛选上下文。
2. 甘特交互：拖拽任务、跨班次调整、锁定任务、查看冲突提示。
3. 任务卡片字段：工序、计划开始、计划结束、计划数量、资源占用、来源（AUTO/MANUAL）。
4. 冲突提示：设备重复占用、员工重复占用、缺料、冻结窗冲突。
5. 保存草稿：保存后写入草稿版本并生成任务变更明细。
6. 撤销/重做：仅对本次编辑会话有效；提交后以变更日志为准。
7. 批量操作前必须展示“影响预览”：影响订单数、可能新增延期数、超负荷变化。
8. 高风险操作支持“短时撤销”或“草稿回退”。

### 3.4 版本中心 `/schedule/versions`
1. 列表列：版本号、状态、基线版本、规则版本、创建人、创建时间、发布时间。
2. 操作：预览、版本差异比对、发布、回滚、导出。
3. 发布前检查：未通过质量门禁时禁用发布按钮并展示失败项。
4. 差异视图：订单级变化、工序级变化、资源占用变化、延期风险变化。
5. 回滚规则：仅 `PUBLISHER/ADMIN` 可执行，且必须二次确认并填写原因。

### 3.5 指令与审批 `/dispatch/commands`
1. 指令类型：INSERT、LOCK、UNLOCK、PRIORITY、FREEZE、UNFREEZE。
2. 发起表单必填：目标订单、生效时间、原因、影响范围。
3. 审批流：显示待审批队列、审批历史、驳回原因。
4. 操作限制：冻结窗内调整、批量锁单、解冻、回滚必须审批后生效。
5. 审计要求：每次提交/审批自动写入 `request_id` 并可追溯。

### 3.6 执行反馈与WIP `/execution/wip`
1. 报工看板：按工序展示计划量、报工量、完成率、偏差值。
2. WIP 批次视图：`wip_lot_id`、来源报工、当前工序、状态、数量、更新时间。
3. 事件时间线：CREATE/TRANSFER/HOLD/RELEASE/SCRAP/CONSUME 全链路展示。
4. 偏差处理：报工偏差超过阈值时显示“触发重排建议”。
5. 跳转能力：可从订单直接跳转到对应 WIP 批次轨迹。

### 3.7 预警中心 `/alerts`
1. 列表列：预警类型、严重度、订单/工序、触发值、阈值、状态、触发时间。
2. 筛选：状态（OPEN/ACKED/CLOSED）、严重度、预警类型、时间窗口。
3. 操作：ACK、CLOSE、指派责任人、关联处理记录。
4. 通知展示：钉钉/邮件/站内消息投递状态与重试次数。
5. 处置SLA：页面显示当前事件是否超时未响应。

### 3.8 审计追溯 `/audit/logs`
1. 查询条件：实体类型、实体ID、操作人、时间范围、request_id。
2. 明细展示：变更前后 JSON、操作原因、客户端IP。
3. 导出：按查询条件导出审计报表（CSV）。
4. 安全限制：导出仅 `AUDITOR/ADMIN` 可见。

### 3.9 主数据管理 `/masterdata`
1. 模块：产品、工艺、路线、设备、设备-工艺能力、员工、技能、班次日历。
2. 批量导入：模板校验 + 预检查 + 提交入库 + 错误明细回传。
3. 生效控制：路线版本、规则版本支持生效时间配置。
4. 防误操作：对影响排产主链路的变更要求二次确认。

### 3.10 同步监控 `/ops/integration`
1. 监控项：入站消息、出站消息、失败重试、补偿任务、同步延迟。
2. 列表列：message_id、source/target、topic、status、retry_count、error_msg、时间戳。
3. 操作：重试、标记忽略、下载原始报文（按权限控制）。
4. 指标卡：5分钟同步成功率、夜间补偿成功率、回写成功率。

### 3.11 计划报表 `/reports/plans`
1. 页签结构：`月计划`、`周计划` 双页签，默认进入月计划。
2. 月计划：按 `doc/offical_resource/2月计划（车间）(1).xls` 的列结构与分组表头显示。
3. 周计划：按 `doc/offical_resource/周计划（3.16-3.22）(1).xlsx` 的列结构显示。
4. 一致性：页面内置周/月共享字段一致性校验；出现差异时显示告警明细。
5. 导出：支持导出当前页签，周计划导出 `xlsx`，月计划导出 `xls`。

### 3.12 仿真平台 `/simulation`
1. 参数：天数、每日接单、场景、随机种子。
2. 页面包含两个子页签：`批量仿真`、`手动模拟`。
3. 批量仿真操作：快进1天/7天/30天、按参数运行、重置仿真、刷新。
4. 手动模拟操作：
   - `虚增生产订单`：一键新增生产订单，直接进入主系统待排池；
   - `推进一天`：在主系统执行当日排产发布与报工推进（不自动新增订单）；
   - `重置模拟`：回滚到手动模拟前完整快照状态。
5. 指标：当前仿真日、销售订单总数、生产订单总数、最新版本。
6. 输出：最近一次运行摘要、仿真事件列表。

## 4. 交互与状态规则
1. 所有写操作必须携带 `request_id`，由前端生成并传递到后端。
2. 冻结窗内写操作必须弹出审批确认，不允许“直接生效”。
3. 列表类页面默认分页：`page=1,page_size=200`，支持最大 `1000`。
4. 时间统一显示 `Asia/Shanghai`，编辑时必须显式展示时区。
5. 页面按钮状态由“权限 + 实体状态 + 审批状态”三者共同决定。
6. 原始数据查看统一走抽屉组件，字段顺序固定为：来源系统 -> 原始字段 -> 更新时间 -> `request_id`。
7. 支持“个人视图保存”：筛选条件、列顺序、列宽、看板布局按用户持久化。
8. 页面内主操作不超过两次点击可达，减少层级菜单跳转。

## 5. 性能与可用性标准
1. 首屏加载目标：`< 2 秒`（常用筛选条件下）。
2. 列表滚动必须无明显卡顿（虚拟滚动）。
3. 甘特拖拽交互响应：`< 150ms`。
4. 指标下钻与抽屉打开响应：`< 300ms`。
5. 失败操作必须有明确错误提示与重试入口。

## 6. 页面到接口映射（最小集）
1. 外部对接接口：`openapi/scheduling-v1.yaml`。
2. 内部页面接口：`openapi/scheduling-internal-v1.yaml`。
3. 报表页接口：`GET /v1/reports/workshop-weekly-plan`、`GET /v1/reports/workshop-monthly-plan` 及对应导出接口。
4. 仿真页接口：`GET/POST /internal/v1/internal/simulation/*`，以及手动模拟接口：
   - `POST /internal/v1/internal/simulation/manual/add-production-order`
   - `POST /internal/v1/internal/simulation/manual/advance-day`
   - `POST /internal/v1/internal/simulation/manual/reset`
5. 指令审批与版本发布接口必须写审计日志。
6. 页面埋点最小字段：`page_code,action_code,request_id,operator,entity_key,timestamp`。

## 7. 验收标准
1. 角色权限矩阵与按钮可见/可操作行为一致。
2. 核心链路“待排池 -> 调度台 -> 版本发布 -> 执行反馈 -> 预警处置”可闭环演示。
3. 所有高风险操作均走审批链并可在审计页追溯。
4. 页面展示字段与数据字典字段一一对应，无歧义口径。
5. 调度驾驶舱可覆盖高频工作流，常用操作集中且可一屏完成。

## 8. 高效操作增强（默认建设）
1. 键盘快捷键：支持 `插单/锁单/重排/发布/搜索` 快捷触发，并提供快捷键帮助弹层。
2. 操作模板：支持常见批量动作模板（如批量顺延、优先级提升）一键套用。
3. 冲突提示可执行：冲突信息必须附带“推荐修复动作”与一键跳转入口。
4. 全局搜索：支持按 `订单号/WIP批次/request_id/设备/员工` 快速定位并下钻。
5. 列配置增强：支持列顺序、列宽、冻结列、显示密度的个性化保存。
6. 自动保存草稿：调度编辑过程自动保存，避免刷新或跳转导致改动丢失。
7. 数据新鲜度提示：每个核心区块显示“最后同步时间”和是否过期。
8. 操作回放面板：展示最近 20 次关键操作，支持在允许窗口内撤销。
9. 异常态统一：加载中、无数据、失败、权限不足统一展示规范并提供重试入口。
10. 移动端轻量策略：移动端仅保留看板、告警处理、审批，不承载重交互甘特编辑。

## 9. 上线保障增强（默认建设）
1. 发布前模拟：提供“模拟发布/影响分析”按钮，先展示影响再允许正式发布。
2. 批量操作防护：所有批量动作必须先预览，并在允许窗口内可撤回。
3. 指标目标线：核心指标卡支持目标线/阈值线显示，异常状态一眼可见。
4. 可执行任务开关：调度台提供“仅看可执行任务”筛选，减少噪音信息。
5. 审批待办前置：将审批待办放在首页显著位置，避免流程阻塞。
6. 防误触保护：关键操作（发布、回滚、解冻）必须二次确认，必要时关键字确认。
7. 外部系统健康可视化：ERP/MES 延迟、失败趋势、重试状态在前端可视化，不依赖查日志。
8. 新手引导模式：首周使用提供场景引导和操作提示，降低上手门槛。

## 10. 工程治理增强（默认建设）
1. 设计令牌与组件标准：颜色、字体、间距、状态色、按钮尺寸统一沉淀为 Design Tokens，核心业务组件（表格、筛选器、抽屉、审批弹窗）必须复用统一实现。
2. 功能开关与灰度发布：高风险功能（自动重排、批量改期、回滚）必须受 Feature Flag 控制，支持按角色/组织灰度启用和紧急关闭。
3. 乐观更新与失败回退：低风险写操作支持前端乐观更新，后端失败时必须自动回滚并提示用户差异结果。
4. 统一错误码映射：建立“错误码 -> 用户提示 -> 建议动作 -> 是否可重试”的前端映射表，避免同类错误多种文案。
5. 埋点与审计联动：页面埋点、接口日志、审计日志统一关联 `request_id`，支持从任意页面一键跳转到审计明细。
6. 可访问性基线：核心页面必须支持键盘可达、焦点高亮、读屏标签与对比度校验，保障高强度班次下的可操作性。
7. 前端性能监控闭环：采集首屏、接口耗时、交互耗时、错误率与白屏率，配置阈值告警并按版本回溯性能变化。
8. E2E 回归最小集：将“待排池 -> 调度台 -> 发布 -> 报工反馈 -> 告警处置”设为强制自动化回归链路，发布前必须通过。
