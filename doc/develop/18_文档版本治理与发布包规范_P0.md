# 文档版本治理与发布包规范（P0，现行）
> 文档属性：现行治理规范。  
> 生效日期：2026-03-22。

## 1. 目的
1. 消除“文档已改但实施口径未更新”的风险。
2. 固化版本发布包与变更审批流程。

## 2. 文档分层
1. 基线文档：`00_现行口径基线_2026-03-21.md`（最高优先级）。
2. 开发规范：`doc/develop/01~20`。
3. 规划规范：`doc/plan/01、03~09、11、12、13`。
4. 历史/模板：仅参考，不可作为实施口径。

## 3. 版本号规则
1. 文档版本：`v{major}.{minor}.{patch}`。
2. 口径变化（破坏性）：`major+1`。
3. 新增非破坏内容：`minor+1`。
4. 文字修订：`patch+1`。

## 4. 变更流程
1. 提交变更申请：说明“变更原因、影响范围、回滚方案”。
2. 评审通过后先改 `doc/develop`，再同步 `doc/plan`。
3. 如涉及口径，必须同步更新基线文档。
4. 发布前执行一致性检查并签字确认。

## 5. 发布包清单（每次上线必备）
1. 需求与范围：PRD、范围边界、验收标准。
2. 技术设计：架构、接口契约、数据库迁移。
3. 数据口径：数据字典、外部数据提供规范。
4. 测试工件：测试报告、缺陷清单、UAT签署。
5. 运维工件：上线步骤、回滚预案、阈值SLA、演练记录。
6. 变更记录：本次版本变更点与影响评估。

## 6. 发布前一致性检查（Gate）
1. 文档之间无冲突字段定义。
2. 接口字段与数据字典一致。
3. 数据库表字段与接口字段一致。
4. 测试用例覆盖变更点。
5. 运维手册包含新变更处置流程。

## 7. 签署机制
1. 业务负责人签署：范围与口径。
2. 技术负责人签署：实现可行性与风险。
3. 测试负责人签署：质量与缺陷状态。
4. 运维负责人签署：上线与回滚可执行性。

## 8. 变更记录模板
1. 变更日期
2. 变更文档列表
3. 变更摘要
4. 影响系统/模块
5. 风险评估
6. 回滚策略
7. 审批结论
