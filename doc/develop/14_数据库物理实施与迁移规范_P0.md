# 数据库物理实施与迁移规范（P0，现行）
> 文档属性：现行开发规范。  
> 生效日期：2026-03-22。

## 1. 目的
1. 将逻辑表设计落地为可执行 DDL、迁移与回滚流程。
2. 明确强外键、逻辑外键、一致性校验的落地方式。

## 2. 目录与命名规范
1. 迁移脚本目录：`db/migration`。
2. 命名规则：`V{yyyyMMddHHmm}__{desc}.sql`。
3. 回滚脚本：`R{yyyyMMddHHmm}__{desc}.sql`（与迁移脚本一一对应）。
4. 禁止手工改线上结构，必须走迁移脚本。

## 3. 表与索引落地规范
1. 主键统一 bigint/雪花或序列（按实现选型固定一种）。
2. 业务唯一键必须物理 unique（不得只写文档）。
3. 高频查询字段必须建索引（参照 `05_数据库设计文档.md` 第3章）。
4. 审计/日志类表按时间字段建组合索引（`created_at` 前缀）。
5. 时间字段统一 `timestamp with time zone`（若数据库策略固定为无时区字段，应用层必须统一Asia/Shanghai）。

## 4. 外键策略

### 4.1 必须物理外键（P0）
1. `schedule_task.version_no -> schedule_version.version_no`
2. `schedule_version.rule_version_no -> rule_param_version.rule_version_no`
3. `dispatch_command_approval.command_id -> dispatch_command.command_id`
4. `equipment_process_capability.equipment_code -> equipment_capacity.equipment_code`
5. `employee_skill.employee_id -> employee.employee_id`
6. `process_route.route_no -> product_route_header.route_no`
7. `process_route.process_code -> process.process_code`
8. `route_step_dependency.route_no -> product_route_header.route_no`
9. `employee_shift_availability.employee_id -> employee.employee_id`

### 4.2 逻辑外键（应用层校验）
1. 所有 `order_no + order_type` 多态关联。
2. `integration_inbox/outbox` 与业务事实表映射。
3. 乱序到达链路（如 `production_material_list_item -> production_order_fact`）。

## 5. 一致性校验任务
1. 每日 `02:00` 执行 `logical_fk_check_job`。
2. 输出 `logical_fk_check_report_YYYYMMDD.json`。
3. 异常分级：
   - `P1`：影响发布
   - `P2`：影响追溯
   - `P3`：可延后修复
4. `P1` 必须在当日发布前清零。

## 6. 分区与归档
1. 大表（`reporting_fact`,`wip_lot_event`,`audit_log`,`notification_delivery_log`）按月分区。
2. 归档策略：保留热数据12个月，冷数据转归档库。
3. 归档前后必须校验行数、校验和、抽样查询一致性。

## 7. 迁移发布流程
1. 开发环境执行迁移并通过单元测试。
2. 预发执行迁移并回放样例数据。
3. 生产执行前先备份数据库和关键配置。
4. 生产执行后必须验证：
   - 关键表可读写
   - 关键索引生效
   - 关键外键生效
   - 迁移版本号记录完整

## 8. 回滚流程
1. 触发条件：迁移失败、性能严重退化、核心流程阻断。
2. 执行顺序：
   - 停写
   - 执行回滚脚本
   - 恢复快照（如必要）
   - 对账验证
3. 回滚后输出 `db_rollback_report_YYYYMMDDHHmm.md`。

## 9. 验收标准
1. DDL 全量可重放（空库可一次建成）。
2. 任意迁移可前滚与回滚。
3. 强外键与唯一约束与文档一致。
4. 一致性校验任务可持续运行并有报告产出。
